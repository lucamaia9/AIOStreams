## VPN Proxy

Location: `vpn-proxy/`

Components:
- **strongSwan** -- IKEv2 IPsec VPN client
- **Squid** -- HTTP/HTTPS proxy on port 8888
- **iptables** -- kill switch (only VPN-matched traffic allowed)

### Dockerfile
Ubuntu 22.04 with strongswan, squid, iptables, rsyslog.

### Startup Flow (`vpn-proxy/start.sh`)
1. Start rsyslog (idempotent -- handles stale PID)
2. Wait for VPN certificate
3. Configure strongSwan IKEv2 (`reauth=no` -- rekey only, no full re-authentication)
4. Boot-race tolerant VPN retry loop (5 attempts, 60s timeout each)
5. Restore Docker embedded DNS (`127.0.0.11`) after tunnel comes up
6. Configure iptables kill switch (IPsec policy matching)
7. Start Squid in foreground (single-process mode: `workers 0`)
8. Background VPN monitor loop (restarts SA if down, exits container if unrecoverable)

### Key Env Knobs
- `VPN_BOOT_MAX_ATTEMPTS=5`
- `VPN_WAIT_TIMEOUT=60`
- `VPN_RETRY_DELAY=10`
- `MONITOR_INITIAL_DELAY=120` (seconds before monitor starts checking)
- `MONITOR_CHECK_INTERVAL=60` (seconds between checks)
- `MONITOR_DOWN_THRESHOLD=3` (consecutive down checks before ipsec restart)
- `MONITOR_RESTART_WAIT=120` (seconds to wait for recovery before container exit)

### strongSwan Config (`vpn-proxy/strongswan.conf`)
Custom config overrides that disable OCSP and CRL revocation checking. The CA
certificate is explicitly trusted and pinned via volume mount, so online
revocation checking provides no security benefit. It was actively harmful
because the kill-switch firewall blocks outbound DNS/HTTP when the tunnel is
down during reauthentication, causing OCSP/CRL fetch failures that lead to
AUTH_FAILED and permanent tunnel death.

### IKE Rekey vs Reauth
The `ipsec.conf` generated by `start.sh` sets `reauth=no`. This means IKE SAs
are refreshed via CREATE_CHILD_SA rekeying (key rotation only, no credential
re-verification) instead of full IKE reauthentication (which re-does EAP +
cert chain + OCSP/CRL). Rekeying avoids the DNS-blocked revocation check
problem entirely and is disruption-free.

### Intermediate Certificates
The AdGuard IKE servers present certificates chained through intermediates
(e.g., ZeroSSL RSA DV SSL CA 2 -> Sectigo Root R46 -> Comodo AAA). strongSwan
needs these intermediates available locally to verify the chain since the
kill-switch firewall prevents fetching them online. All four lanes (Helsinki,
Amsterdam, London, NYC) now mount the ZeroSSL RSA DV SSL CA 2 and Sectigo
Root R46 intermediates alongside their lane-specific Comodo CA cert.

### VPN Monitor Loop
Safety net for cases where strongSwan's `dpdaction=restart` can't recover
(e.g., AUTH_FAILED, server cert rotation, persistent IKE_SA_INIT failure).
The loop runs in the background:
1. Checks `ipsec status` every 60s
2. If SA is down for 3 consecutive checks, runs `ipsec restart`
3. If tunnel doesn't recover within 120s after restart, kills PID 1 so
   Docker's `restart: unless-stopped` recreates the container from scratch

### Squid Config (`vpn-proxy/squid.conf`)
- Port: 8888
- ACLs: localhost + 172.22.0.0/16
- Single-process mode (`workers 0`) to prevent zombie children

### Prowlarr Per-Indexer Proxy (TRaSH Guides Pattern)
Prowlarr uses per-indexer proxy assignment via tags instead of broad app-wide proxying. Reference: `https://trash-guides.info/Prowlarr/prowlarr-setup-proxy/`

- **EZTV** (tag `flaresolverr-helsinki`, ID 8):
  - `Http-EZTV-Helsinki` — Helsinki HTTP proxy for plain requests
  - `FlareSolverr-EZTV-Helsinki` — Helsinki-backed FlareSolverr for CF solves
  - Custom Cardigann definition `prowlarr/config/Definitions/Custom/eztv_douglas.yml`
  - EZTV is geo-blocked from Oracle Cloud IPs (HTTP 451 without Helsinki VPN)
- **Slow indexers** (tag `amsterdam-lane`, ID 9):
  - `Http-Amsterdam` — Amsterdam HTTP proxy for non-CF requests
  - Mikan, The Pirate Bay, Torrent9 time out frequently from Oracle Cloud
  - All 3 still carry tag `flaresolver` (ID 1) for FlareSolverr fallback via `FlareSolverr-Prowlarr` (which itself egresses through Amsterdam VPN)
- **All other Prowlarr indexers** (tag `flaresolver`, ID 1):
  - `FlareSolverr-Prowlarr` — Amsterdam-backed FlareSolverr (see Service Mapping above)

## DNS Fix (2026-03-18)

Root cause of Torrentio proxy failures was DNS resolution inside `adguard-proxy-helsinki`:
- Container could reach IP literals over VPN but hostname lookups via strongSwan-pushed resolver timed out
- Fix: restore Docker embedded DNS (`127.0.0.11`) after tunnel comes up
- Squid configured with `dns_v4_first on` to use that resolver

## Historical Fixes

### AdGuard Restart-Loop Fix (2026-03-01)
- Root cause: `rsyslogd` stale PID file after manual restart
- Fix: idempotent rsyslog startup in `vpn-proxy/start.sh`
- Rolled out to all three lanes

### Zombie Prevention (2026-03-01)
- Root cause: Squid master/worker pattern produced defunct children
- Fix: `workers 0` in `squid.conf` for single-process mode
- Host zombie scan returns empty after fix

### Boot Stability Hardening (2026-03-01)
- Root cause: VPN establishment race with strict health checks
- Fix: bounded retry loop, local health check instead of external endpoint, relaxed startup window

### AdGuard Memory Limit Fix (2026-04-22)
- Root cause: 96MB memory limit too tight for overlay2 filesystem + cgroup v2 strict enforcement
  - When container restarts (e.g., after VPN drop), Docker writes DNS config to overlay filesystem
  - This requires memory for page cache, inode structures, metadata
  - With only 96MB, kernel returns ENOSPC ("no space left on device") despite host having free disk
  - Only Helsinki/NYC crashed initially; all 4 were at risk
- Fix: Increased `mem_limit` from 96MB to 256MB, `pids_limit` from 128 to 256
- Why not detected earlier: `docker ps` showed stale "healthy" status for crashed containers (Docker daemon bug #32413)
- Restart policy didn't recover: ExitCode 1 before 10s uptime means restart policy may not engage
- References: FlareSolverr containers use 512-768MB; 96MB was based on historical usage (~18MB) but didn't account for startup overhead

### Byparr Removal (2026-03-02)
- Removed `byparr`, `byparr-jackett`, `byparr-prowlarr` from compose
- Reason: `BYPARR_MODE=off` in Comet, so byparr was not in traffic path
- If needed again, must be reintroduced in compose (or opt-in profile)
